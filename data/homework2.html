The question

Here is a function that makes a bar graph with three bars.

fun bar-graph(num-a :: Number, num-b :: Number, num-c :: Number) -> Image:
  doc: ``` given three numbers, produces a bar graph with those heights
       cannot take a number greater than 26 ```
  underlay-xy(
    underlay-xy(
      underlay-xy(
        rectangle(80, 80, "solid", "tan"),
        8 + (0 * 25), 80 - (3 * num-a),
        rectangle(15, 3 * num-a, "solid", "red")),
      8 + (1 * 25), 80 - (3 * num-b),
      rectangle(15, 3 * num-b, "solid", "blue")),
    8 + (2 * 25), 80 - (3 * num-c),
    rectangle(15, 3 * num-c, "solid", "green"))
end

For example, bar-graph(20, 15, 24) produces this image:

  1. In a code block (see Helpful Things), explain the restriction of the input numbers to 26 pixels. It may be helpful to copy the code into CPO.
  2. Write a function bar-graph-clean that cleans up bar-graph by introducing helper functions or constants where appropriate.
    Note: bar-graph-clean should have the exact same functionality as bar-graph. If you run bar-graph(20, 15, 24) == bar-graph-clean(20, 15, 24), it should produce the boolean true.
    Note: We are not asking you to remove the 26 pixel restriction here.
  3. Write a function bar-graph-general that takes in three arguments (just like bar-graph) and produces a bar graph regardless of what those three numbers are. bar-graph-general(50, 2, 120) should run perfectly fine, for example.

The question

Here is a function that computes the cost of a pizza based on a single topping and whether or not the pizza was delivered.

fun order-cost-orig(topping :: String, delivered :: Boolean) -> Number:
  doc: ``` Adds a 15% tax to the cost of a pizza based on one topping,
  adding a (untaxed) delivery cost if appropriate ```
  if topping == "onion":
    if delivered:
      12.50 + (12.50 * 0.15) + 3
    else:
      12.50 + (12.50 * 0.15)
    end
  else if topping == "pepperoni":
    if delivered:
      14 + (14 * 0.15) + 3
    else:
      14 + (14 * 0.15)
    end
  else if topping == "none":
    if delivered:
      11 + (11 * 0.15) + 3
    else:
      11 + (11 * 0.15)
    end
  end
end

  1. In a code block, list a few of the limitations of this function. Discuss its structure and clarity, as well as any functionality drawbacks there may be.
    In addition, discuss how the function would need to change if a drink was also being bought. Would it be simple or complicated to incorporate, and why?
    Your answer should look like this:

    #|
- One disadvantage is that ...
- ...
- Adding a drink cost would be ...
|#

  2. Writing a cleaner version of order-cost-orig
    Using helper functions and constants where appropriate, write a function order-cost that takes in a topping string (either “onion” or “pepperoni”) and a delivery boolean, and computes the price of the pizza.
    Note: The functionality of order-cost should not change. order-cost and order-cost-orig should do the same thing with the same inputs.
    Note: This question is asking only for improvement on the design of order-cost, not any functionality limitations you may have discussed in question 1.

  3. Write a few test cases for order-cost. You may or may not need to test all 6 possibilities, depending on how you structured your code. We will be lenient in grading tests for this assignment as long as you have put effort into creating valid and thorough tests.
